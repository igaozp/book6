# IPv6 中的过时特性

本章列出了许多可以被认为过时或未使用的 IPv6 特性，即使在某些情况下相关的 RFC 尚未正式废弃。建议读者在新部署中忽略这些技术，并考虑从现有部署中删除它们。

## 移动 IPv6

移动 IPv6 是一项在 IPv6 协议开发的早期就嵌入其中的技术。然而，基于零实现，该技术仅存在于教科书和 IETF 标准中。由于没有当前的实现，并且当前的标准极其陈旧，移动 IPv6 实际上已经过时，不应在培训中教授或作为 IPv6 特性集的一部分提及。以下是这些 IETF 标准的列表：

- [RFC 6275](https://www.rfc-editor.org/rfc/rfc6275.html) - IPv6 中的移动性支持最后更新于 2011 年。它是涵盖基本操作、安全功能、移动性操作和数据包结构的主要标准。
- [RFC 4283](https://www.rfc-editor.org/rfc/rfc4283.html) - 移动 IPv6（MIPv6）的移动节点标识符选项最后更新于 2005 年。它主要涵盖移动性头的所有功能，并且必须有一种机制，使移动实体可以使用其他标识符进行识别。
- [RFC 4285](https://www.rfc-editor.org/info/rfc4285) 移动 IPv6 的认证协议最后更新于 2006 年。这是使用包含在这些消息中的移动性消息认证选项来保护移动节点和归属代理之间的绑定更新和绑定确认消息的解决方案。

## 站点本地单播地址

站点本地单播地址在 IPv6 中占据着一个奇特的位置。它最初被提议提供私有 IPv6 地址，类似于 [RFC 1918](https://www.rfc-editor.org/rfc/rfc1918.html)。站点本地地址最初在 [RFC 3513](https://www.rfc-editor.org/rfc/rfc3513.html#section-2.5.6) 的第 2.5.6 节中为这种特殊用途定义。由于以下原因，问题很快出现：

- 地址的歧义性，意味着诸如 `fec0::1` 之类的地址可以存在于多个站点中，并且地址本身不包含它所属站点的任何指示。这给应用程序的开发人员、路由器的设计人员和网络管理人员带来了痛苦。
- 站点的模糊定义与地址的歧义性相结合，意味着如果站点合并（例如，当两家公司合并时）或以其他方式交叉路由，要么
  1. 必须重新编号主机，要么
  2. 必须添加区域标识符以区分节点，例如 `fec0::1234:5678:9abc%1`。这需要不存在的软件和操作支持。

### 已弃用但未消失

站点本地地址很快被 [RFC 3879](https://www.rfc-editor.org/rfc/rfc3879) 弃用。然而，即使有了这种弃用，文档中仍然存在痕迹，并且造成了严重的混淆。[RFC 4291](https://www.rfc-editor.org/rfc/rfc4291.html#section-2.5.7) 提到了这种弃用。

- RFC 3879 还声明前缀 fec0::/10"除非未来的 IETF 标准行动，否则不得重新分配用于其他用途。"鉴于此，它不能被重新吸收到 IANA 全球单播地址空间中。
- 它还在 [RFC 6724](https://www.rfc-editor.org/rfc/rfc6724.html) 中概述的 IPv6 前缀策略表中作为单独的前缀列出，标签为 11，优先级为 1。
- RFC 3879 和 RFC 4291 都声明现有的实现和部署可以继续使用站点本地前缀。

尽管如此，站点本地单播地址实际上已经过时，不应在培训中教授或作为 IPv6 特性集的一部分提及。

## 扩展头的问题

在实践中，添加新的扩展头（包括新的逐跳选项）已被证明非常困难，因为安全性和拒绝服务问题意味着它们经常在网络边界被阻止。它们还使负载均衡器等技术难以快速识别数据包中的传输头。（更多详细信息请参见 [2. 扩展头和选项](../02.%20IPv6%20Basic%20Technology/Extension%20headers%20and%20options.md) 的末尾。）

特别是，路由器警报已被弃用用于未来的协议
  \[[RFC9805](https://www.rfc-editor.org/info/rfc9805)\]。
