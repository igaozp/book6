# 安全性

安全性在总体上具有日益增长的重要性，IP 协议一直是安全研究和开发的重要领域。大多数 IPv4 实践仍然适用于 IPv6。但对于第一跳的某些方面以及 IPv6 中显著不同的扩展头来说，存在例外情况。分布式地址获取（SLAAC，[2. 自动配置](../02.%20IPv6%20Basic%20Technology/Auto-configuration.md)）带来了额外的安全挑战。每个主机的多个地址改善了隐私性，但也并非没有复杂性。扩展头为 IPv6 提供了极大的灵活性和可扩展性，但也可能被滥用，导致需要额外的安全预防措施。

最初，人们期望端到端加密（加密和认证）将成为 IPv6 的强制性部分（IPsec，[RFC 4301](https://www.rfc-editor.org/info/rfc4301) 和 SEND，[RFC 3971](https://www.rfc-editor.org/info/rfc3971)）。但这被证明是不现实的，因此加密在网络层被接受为可选的，就像 IPv4 一样。与此同时，加密在传输层或应用层已经变得广泛普及。

IPv6 旨在恢复网络层的端到端连接性。因此，IPv6 安全性绝不依赖于网络地址转换的存在。IPv6 没有标准化的 NAT66，甚至网络前缀转换（NPTv6，[RFC 6296](https://www.rfc-editor.org/info/rfc6296)）也很少使用。NAT 或 NPTv6 在网络边界提供的安全保护充其量是微弱的，因此这不被视为缺陷。IPv6 边界安全的正常方法是防火墙；大多数防火墙产品同时支持 IPv6 和 IPv4。拓扑隐藏将在本章后面的章节中讨论。

如今，安全领域的"零信任"方法倾向于将重点从周边保护转移到对所有流量（包括任何周边内部的流量）的认证和加密。如果这种方法成功，一些企业可能会选择在未来减少防火墙的作用。IPv6 为这一变化做好了充分准备。

一个良好的设计和策略规则是：在双栈部署中（这是迄今为止 IPv6 部署的最大比例），IPv4 和 IPv6 的安全策略应该匹配，以确保操作和用户体验的一致性。在纯 IPv6 部署中，策略的实施应该源自整体网络安全策略，同时考虑到可能需要从传统 IP 进行调整的协议规范（即 IPv4 和 IPv6 之间 ICMP 处理的差异）。

[RFC 9099](https://www.rfc-editor.org/info/rfc9099) 提供了 IPv6 安全性的良好概述。这是一个很好的参考资料库，涉及各种 IPv6 安全方面的许多文档。
