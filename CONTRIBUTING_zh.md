# book6 - 一本协作的 IPv6 书籍

本文件描述如何做出贡献。

贡献者应是自我声明的 IPv6 从业者。除了有效的 GitHub 账户外，没有其他准入要求。

贡献（以 GitHub markdown 格式）应符合约定的[目录](Contents.md)。它们应是事实性的，并向技术读者讲授 IPv6 的特定方面。尽可能引用 RFC 和其他免费提供的文档。

当人们部署或使用 IPv6 时存在替代方案和选择时，应客观地呈现这些选择，如果可能的话附上事实性的利弊。可以提出合理的建议（例如，"X 通常比 Y 更安全"），但应避免强烈的个人观点（例如，"NAT66 永远不是正确的答案"）。

贡献者可以编辑自己和他人的贡献。但是，重大更改应首先使用问题追踪器进行讨论。对于拼写错误、语法问题或小修正则不需要这样做。必要时将使用 GitHub PR（即，当添加或更改需要审查时）。目标是在内容上达成 IETF 风格的粗略共识。一个典型的决胜论据是"我已经部署了这个，它运行良好。"

通常，一个贡献将是一个完整的章节。基于目录，文件有一个命名约定，以便我们可以编写脚本从代码仓库生成整本书。请参阅[基础章节模板](99.%20Chapter%20Template/99.%20Chapter%20Template.md)并仔细遵循它。该章还包含使用 GitHub markdown 的说明。请在其章节文件夹内为每个小节创建单独的文件。有关详细说明，请参阅章节模板本身。如果您不遵循这些说明，事情会变得混乱，需要手动整理。

如果您更喜欢提交包含纯文本的 PR，最终会有人添加 markdown，但可能会有延迟。请不要提交其他格式。

请注意，贡献**必须**是原创写作，除非贡献者有合法权利根据约定的许可证提交现有材料（参见 [LICENSE.md](LICENSE.md)）。

如果我们直接引用 RFC 的内容，它们**必须**是可以被视为相关 [IETF Trust FAQ](https://trustee.ietf.org/about/faq/#reproducing-rfcs) 中提到的"合理使用"的小摘录，在"我是否被允许复制 RFC 的摘录？"下。但是，RFC 的*修改*摘录是不允许的。

（为避免疑义，如果*您*个人是原始材料的版权持有者，*您*可以在此贡献它，并根据 [LICENSE.md](LICENSE.md) 修改它，即使它也已贡献给 RFC。）

我们将任命两三个人作为顶级编辑，他们可以宣布共识并批准 PR。

语言是英语，最好使用美式拼写。请保持句子合理简短和简单。避免使用对母语非英语的人可能不清楚的语言。避免在专业写作中不合适的术语。如有疑问，请参考已存档的 [NIST 包容性语言指南](https://web.archive.org/web/20250203031433/https://nvlpubs.nist.gov/nistpubs/ir/2021/NIST.IR.8366.pdf)。如果添加图表，请考虑视障人士的可访问性，并确保提供替代文本。

任何翻译成其他语言的版本必须采用与原始版本相同的开源许可证，并且必须免费提供。
